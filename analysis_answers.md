Task 1.2: 1 What happens when the planner produces steps that depend on each other (e.g., step 3 needs the result of step 1)? How does your architecture handle this?
Dependent steps: Your architecture handles this implicitly through step_results — each step's result gets appended to a shared list, and the supervisor/executing node can reference prior results when constructing its action (e.g. the MCP calculation step uses the revenue figure pulled from step_results[0]). There's no explicit dependency graph — it relies on the planner ordering steps correctly (retrieval before calculation) and on step_results being visible to later steps. If step 3 needed step 2's result but the planner ordered them wrong, this would break silently — worth naming as a limitation.

Task 1.2: 2 Would a replanning step after each execution improve or hurt performance for this use case? Justify with an example.
Replanning: Could go either way depending on the example — for this simple 2-3 step CAGR-style task, replanning after each step would add LLM calls/latency without benefit since the plan is usually correct upfront and steps are independent enough. But for a query where a retrieval result changes downstream needs (e.g. "find the segment with the best margin, then calculate its 5-year growth" — you don't know which segment until step 1 finishes), replanning would help since the fixed plan can't specify the right next step in advance.
